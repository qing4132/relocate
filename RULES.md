# 原则

这是一个业余维护的"移民 / 移居路径"参考网站。

本文件不是硬章程，是基调。冲突时：现实优先于文字，文字优先于记忆。

---

## § 1 · 定位

1.1 **永远非盈利**。本项目不接广告、不接赞助、不接中介返点、不做付费墙、不做会员订阅、不做任何形态的商业化。以后也不做。

1.2 **仅供参考**。网站上展示的每一条签证规则、税率、成本、时间线都**不保证**最新、不保证正确、不保证完整。读者在依据本站信息做任何实际动作前，必须到对应国家的政府官网亲自核对。

1.3 **不为任何人负责**。本项目不对任何读者的移民结果、经济损失、身份异动、家庭变化负责。这不是免责声明话术，是项目形态承诺——因此永远不做代办、陪跑、保过、一对一咨询。

1.4 **"仅供参考 · 非官方"字样须在每个页面首屏可见**。不得藏进服务条款、不得缩到页脚 12 号灰字。

---

## § 2 · 信息质量（尽力而为）

2.1 **有数字或具体规则的陈述，尽量附官方来源链接**。做不到时允许二手整理（律所、博客、论坛）作为初步线索，但必须在文中显式标"二手"。

2.2 **原件能存档就存**，存到 `data/archive/<authority>/<docId>/<YYYY-MM-DD>/` 下。不强制、不 block 发布。目的是方便以后复核，不是向读者证明权威。

2.3 **不凭空编数字**。这一条是底线。

- "官方没说"写"资料暂缺"或"未公开"，**不写**"大约"、"参考行业均值"、"与 X 相近"
- LLM 辅助**整理文字**可以；LLM 生成**事实性数字**不行
- 整理过程中自己不确定的地方，在页面上写"不确定"，不强行给一个数

2.4 **过时是常态**。发现过时 / 错误 → 改之，commit message 写清"从哪查出来的、改成了什么"；不追究原录入者；不静默改写历史。

2.5 **复核按能力来**。理想是每 180 天重看一次所有条目；现实能走到哪算哪。到期未复核的条目可以在页面上标"最后核对于 YYYY-MM-DD"，由读者自判是否过时。**不因为复核落后就假装信息还新**。

---

## § 3 · 边界

3.1 **永远不做**：

- 广告 / 赞助 / 中介合作 / 返利返点
- "AI 移民顾问"聊天式产品（LLM 助写文字不算此列）
- 代办 / 陪跑 / 保过 / 一对一移民咨询
- 用户增长黑客（分享得 VIP、邀请返现、积分解锁、签到打卡）
- 付费墙、登录墙、地区墙
- 用"看起来权威"的视觉伪装非官方性质

3.2 **不征求外部用户试用、不做主动推广**。项目停留在"自用 + 顺手公开"的形态。有人自然找到并觉得有用，皆大欢喜；没人找到，也不亏。

3.3 **不把自己当作任何特定人群的救生筏**。目标读者是"想了解移民路径的中文读者"，仅此。不做年龄 / 资产 / 学历分层，不做"适合 XX 的移民方案"这类细分栏目。

---

## § 4 · 写作与视觉

4.1 **编辑风、文本密集、克制留白**。Serif 标题 / Sans 正文。参考严肃长读物与研究报告。

4.2 **不用营销话术、不用装饰 emoji**。禁止的词：推荐、最佳、首选、宜居、一站式、AI 驱动、智能、抓紧、限时。

4.3 **每个数字后面附一句人话解读**。裸数字不上页面——不是因为"体验好"，是因为裸数字在"仅供参考"语境下最容易被误读为结论。

4.4 **不替读者做选择**。不说"这条路最好"、"建议选 XX"；呈现信息、列出差异、停在那里。要不要去、选哪条，是读者的事。

4.5 **修改前先读一遍改前的内容**，提交前再读一遍改后的内容。两次都能一眼看懂再提交。

---

## § 5 · Commit 与历史

5.1 每次 commit message 写清：**改了什么、为什么改**。"小修"、"update" 不是合格 message。

5.2 错误修正不悄悄改。发现老内容错了，改它，在 commit message 里说"发现于 YYYY-MM-DD，原内容 X，已改为 Y，原因 Z"。

5.3 不删除历史。`archive/` 里的试错留痕永远保留，哪怕它与现定位冲突——**它是"我们试过什么、为什么放弃"的证据**。

## § 6 · 这份文件本身

本文件会随项目现实演进而改。改它本身没有额外仪式——一个 commit，message 说清为什么，完事。

早期版本（2026-04-19 之前）曾是一份硬章程，定义了付费年费模型、180 天强制复核、字节级快照必填、一手源独立 PR 登记、Phase 0 dogfood 验收等一整套严格约束。现在那些都放宽了——不是因为发现它们错了，是因为**项目形态换了**：当目标只是"一个业余维护、仅供参考、不盈利、不担责的移民路径资料站"，原来那套章程是为另一种形态（一个要长期稳定输出、要靠付费反哺复核工作量、要作为用户真实决策依据的产品）准备的，在当前形态下是过度工程。

## § 0 · 元规则

0.1 **本文件优先于任何其他约定**。任何 AGENTS.md、README 注记、聊天记录、会议口头共识、辅助 AI 的建议——冲突时以 RULES.md 为准。

0.2 **合理化解释不能稀释规则**。如果你发现自己在为"这次破例"写一段很长的理由——停手，按规则办。长理由本身就是违反信号。

0.3 **这是负面清单，不是路线图**。遵守它不会让产品成功；违反它会让产品必然失败。不做对一件事只是慢，做错一件事会让前面的所有工作作废。

0.4 **规则可以被推翻，但推翻要付代价**。代价包括：公开复盘、在项目对外文档留记录、在产品内把失败披露给用户。零代价的"破例"一律拒绝。

0.5 **本项目从零开始，不继承任何外部资产**。历史上任何代码、数据、品牌、域名、用户群、想法草稿，到此为止都视为不存在。"之前那边有现成的"不是复用的理由——是必须独立重做的理由。

## § 1 · 态度（第一性）

1.1 **"看起来差不多" ≠ 正确**。每一次修改提交前，逐项对照输入要求核对输出。不匹配就不提交，哪怕只差一个像素、一个字段、一句翻译。

1.2 **宁可不做，也不凑活**。任何"先这么着、以后再改"的决定，要么当场拒绝，要么在同一 commit 里写下可验证的补救期限。没有期限的 TODO 不予入库。

1.3 **用户信任一次性**。任何让用户看到"估算值"却没显式标注其性质与来源的行为，就是欺诈。不接受"但我们在代码注释里写了"——UI 上看不到的诚实等于不诚实。

1.4 **犯错之后，先承接错误，再谈改正**。禁止用"流程改进"、"效率提升"的话术快速翻篇。反复犯同一个错 = 没把前一次的教训当回事，是态度问题，不是能力问题。

1.5 **效率不是借口**。选择永远是"最正确的方案"，不是"最快写出来的方案"。实现成本不构成降低品质的理由。

1.6 **目标是"正确完成"，不是"快速完成"**。

## § 2 · 数据：从生到死

### 2.1 · 出生

2.1.1 产品里的每一个对外可见的数值字段，必须来自 `SOURCES.md` 白名单列出的一手源。新增源需要先在独立 PR 里更新 `SOURCES.md` 并说明采纳理由，且**不得在同一 PR 中使用它**。

2.1.2 **永久禁止的输入**：
- Numbeo / Expatistan / Nomadlist / 一切 UGC 聚合站
- 一切第三方移民中介或律所内容站
- 一切 LLM 生成文本（GPT / Claude / Gemini / 本地模型皆然），包括"AI 辅助填充缺失值"这一整类动作
- 一切"根据类似城市推断"、"同国家均值"、"历史比例外推"的推算值
- 一切"大致"、"约等于"、"参考行业均值"的口语估计

这份禁令不随模型能力升级而放宽。LLM 再强也不是一手源。

2.1.3 **政策类规则（签证、税务、身份）只接受政府官方机构**的官网、官方 PDF、法规原文。第三方聚合器（即使是知名律所、知名 SaaS、知名论坛）都不是源。

2.1.4 **字段的"存在"与"取值"同样受一手源约束**。一个字段能否进入产品的数据 schema，前置条件是至少有一个白名单一手源能稳定提供该字段。不得为了产品叙述完整性先设字段、再找来源——字段集是一手源集合的子集，不等式不可反向。

例：用户可能想看"某公司对中文候选人的 offer 年薪范围"。官方一手源不提供此类数据，因此该字段**不存在于 schema**，也不会以任何形式出现在 UI。替代做法是展示"该职业 / 学历 / 经验区间的国家级收入中位数（源：该国统计局）"，由用户自行推断。此规则同时消解了"关键字段为 null 怎么办"的困局：没有一手源就不设字段，字段一旦存在就必有源。

### 2.2 · 形状

2.2.1 任何写入数据库或产品 JSON 的字段，必须携带五元组：

```
{ value, sourceURL, fetchedAt, rawQuote, snapshotRef }
```

- `value` 允许为 `null`；不得是"基于 X 推算的 Y"
- `sourceURL` 必须是一手源的永久地址（不是搜索结果页、不是 Wayback Machine 的快照）
- `fetchedAt` 必须是真实抓取/录入那天的日期，禁止整月、整季度这类虚指
- `rawQuote` 是官方页面或 PDF 的节选，目的是让 6 个月后复核的人在 60 秒内原地核对
- `snapshotRef` 必须指向 `data/archive/` 下一份真实存在的原件快照（见 §2.7）；rawQuote 必须能在该快照中找到。**无快照 = 无字段**

缺任何一项的字段不得落地。

2.2.2 **空值合法，估算违法**。官方源没有就写 `null`。不允许用邻近地区、历史趋势、"合理范围"替代。

### 2.3 · 流通

2.3.1 **每个数据源是独立目录，互不依赖**（`data/sources/<source>/`）。源脚本只产出该源的原始 JSON/CSV，不得直接写产品数据文件或产品数据库表。

2.3.2 **`data/build.mjs` 是唯一写入产品数据的闸门**。任何脚本、任何手工 `jq`、任何编辑器里的"就改一下 JSON"都不得绕过它。绕过一次就污染一次，污染不可逆。CI 应静态检查这条约束。

2.3.3 **源优先级由 `SOURCES.md` 显式声明**。仲裁层按优先级取值，**不做加权平均、不做几何平均、不做任何"聪明的融合"**。融合就是引入估算。

### 2.4 · 展示

2.4.1 每个对外展示的数值，必须在至少一个可达交互（点击 / 悬停 / 详情面板）里暴露 `sourceURL + fetchedAt`。不得把"数据溯源"当可选设置项藏起来。

2.4.2 **过期显式标注**。各类数据最长有效期：
- 政策规则（签证、税务、身份）：180 天（强制复核规则见 §5.2）
- 房租 / 生活成本 / 汇率：90 天
- 薪资（官方披露类）：365 天或自源发布下一期起失效，以先到者为准
- 治理 / 权利类年度指数：545 天（~ 18 个月）

超期必须在 UI 上打"数据过期·待复核"徽章。过期数据不是 bug，是必然；假装不过期才是 bug。

2.4.3 **"暂无"是正常状态**。数据缺失时 UI 展示"该字段暂无可信来源"，不得展示 0，不得展示"—"，不得用邻近值填充视觉空白。

### 2.5 · 死亡

2.5.1 当一个源被发现系统性偏差，该源的**所有历史字段必须被标记为污染并从产品退役**，不得"只改未来数据"继续使用旧值。

2.5.2 退役事件必须在 `data/retirements.log` 留下记录（日期 / 源 / 偏差描述 / 受影响字段数 / 受影响快照路径列表 / 替换源 / 处置 commit）。该日志对外公开可见，作为数据诚实度的证据。

### 2.6 · 叙事数据（对外文本里的数字）

2.6.1 **叙事文本里的每一个数字也是数据**，同样适用 §2.1-§2.2。项目时长、commit 数、用户数、MRR、访问量——落笔前先查源（git log / 后台 / DB），禁止凭语感写。

2.6.2 写"数据诚实"的文档时自己伪造数字，是本项目最严重的自打嘴巴。一旦发生，立刻发公开勘误，不在 PR 里静默修改。

### 2.7 · 存档（原件字节级快照）

2.7.1 **原件存档是硬要求**。每次从一手源取回的原件（PDF / HTML / CSV / XLSX / JSON API 响应等）必须以**原样字节**存档于 `data/archive/` 下，不转码、不压缩、不清洗。存档路径规范：

```
data/archive/<authority>/<doc-id>/<YYYY-MM-DD>/<sha256>.<ext>
data/archive/<authority>/<doc-id>/<YYYY-MM-DD>/<sha256>.meta.json
```

- `<authority>`：一手源机构代码，形如 `jp/moj`、`jp/mhlw`、`jp/nta`，在 `SOURCES.md` 统一声明
- `<doc-id>`：人类指定的稳定标识符（例 `hsp-scoring-table`），不由官方 URL 推导——URL 可变，标识符不变
- `<YYYY-MM-DD>`：本次抓取的日期（UTC，同 `fetchedAt` 的日期部分）
- `<sha256>`：原件内容的 SHA-256（小写十六进制，全长 64 位或前 16 位缩略二选一，由施工阶段统一定义后写入施工规范文档）
- `<ext>`：原件扩展名，按实际 Content-Type 定

2.7.2 **不可变**。存档文件一旦 commit 不得覆盖、不得 `git rebase` 重写历史、不得删除。同一 doc-id 的新版本**以新日期新路径新增一份**，而非覆盖旧版。抓取错误、文件损坏等情形，在伴生 `.meta.json` 里标 `invalid: true` 并记录原因，文件本体仍然保留。

2.7.3 **伴生元数据必须完整**。每份存档必须有一份同目录同 sha 前缀的 `.meta.json`，至少含：

- `fetchedAt`：ISO-8601 精确到秒的 UTC 时间戳
- `sourceURL`：本次请求的完整 URL（包含 query string）
- `httpResponseHeaders`：至少包含 `content-type`、`last-modified`（如有）、`etag`（如有）、`content-language`（如有）
- `sha256`：全长 64 位 SHA-256
- `bytes`：字节数
- `fetcher`：抓取者标识（脚本名+版本或人名+日期）
- `authority`、`docId`、`language`、`mediaType`：冗余但便于检索

缺任何一项的存档不得 commit。

2.7.4 **字段引用具体快照**。产品数据中的每个 `snapshotRef` 必须指向 `data/archive/` 下真实存在的文件，sha256 必须与路径中的 `<sha256>` 和 `.meta.json.sha256` 全部一致。CI 静态检查：`value → snapshotRef → 文件存在 → sha256 匹配 → .meta.json 有效 → rawQuote 能在快照内定位` 全链路。断一环 CI 红。

2.7.5 **存储策略**。二进制原件以 Git LFS 管理，与源码同仓。**不得以"体积大"、"克隆慢"为由不存**。仓库真正体积失控（规模阈值在施工期实测后在施工规范文档定义）时，才论证分仓——但分仓后新仓仍遵 §2.7 全部约束。

2.7.6 **再分发合规**。每个源必须在 `SOURCES.md` 明确标注该国政府文档的再分发条款：

- **公有领域 / 明确允许再分发** → 可存档进公开仓库
- **仅允许非商业再分发** → 本项目 Phase 0 为非商业自用，可存档；商业化部分（§7.3 年费服务）上线前必须重审，将来可能需要转为私有存档或将该国退出覆盖
- **禁止再分发** → 禁止存档进公开仓库；如确实需要该源，该国待 §3.8 重新审视
- **条款不明** → 同“禁止”处理，不得假设默认开放

再分发条款变更时，本项目须在 30 天内响应（调整录入路径或退役相关存档）。

2.7.7 **指定日期的可回放性**。给定任意历史日期 T，项目必须能还原出"在 T 时点，产品里每个字段引用的快照版本"。这是对 §5.2 复核机制的闭环：复核不只是认证什么没变，还要有能力证明什么变了 / 什么没变 / 对老用户影响几何。

## § 3 · 产品定位与功能边界

3.1 **产品是决策引擎，不是仪表盘**。输出的是"可执行的路径 + 概率 + 时间线 + 真实成本"，不是数据表格。任何"再加一个比较维度"、"再多展示几个指标"、"搞一个炫酷 dashboard"的提议，默认不做。

3.2 **v0 范围必须显式锁定**（具体数字在 Phase 0 结束时确定并写入 `scope.md`），锁定后不接受"再加一个国家就上线"。

3.3 **永久不做清单**：
- 全球覆盖假象（超过 v0 锁定范围的城市/国家一律标"暂不覆盖"）
- AI 问答 / 对话式签证助手
- 数字游民签证专题（非核心用户）
- 退休移民专题（非核心用户）
- 跨境投资建议（合规雷区）
- 移民中介直营（会稀释公信力）
- 任何形式的"AI 生成的城市介绍 / 签证解读"
- 任何"分享得 VIP"、"邀请返现"、"积分解锁"类增长黑客机制

3.4 **明确说"我们还没做"比假装覆盖更体面**。UI 在不支持的场景下显示"该场景暂不在我们覆盖范围"，不得展示空白表或默认值。

3.5 **每一个"也顺便服务一下那类人"的拓展都稀释主角定义**。非目标用户可以用，但产品不为他们做任何适配。目标用户画像（产品适配对象的权威定义）：**持中国大陆护照、中文母语、25–45 岁、本科及以上、严肃考虑换国家继续从事专业工作**。同时满足这五条才是 v0 的适配对象。少一条的人可以用产品，但产品不做任何向他们的倾斜。

**收入、资产量级不在画像里**。产品卖信息不卖服务；同一份日本 HSP 积分规则对年薪 200 万的人和年薪 20 万的人都是相同的信息，产品不按资金划分用户等级。

3.6 **v0 的硬前置：用户持中国大陆护照**。这不是用户输入项，是产品层面的前置常量。国籍因此不出现在 v0 的用户问答流中。扩展到其他华人护照（港/澳/台/已入籍第一代）须在 v0 通过 §9.1 的 dogfood 验收**之后**，以独立 Phase 的形式启动，并在启动前修改本条。

3.7 **产品不替用户做决策**。

3.7.1 产品是**冷的信息整合工具**，把用户推到决策临界点，**不跨过那一点**。跨过的那一步——"要不要去"——永远是用户的动作。产品因此不做任何形态的"包办"、"陪跑"、"保过"、"咨询"服务；用户因此也不能反过来要求产品替他判断。

3.7.2 **文案禁用替用户判断或制造紧迫感的措辞**：禁止出现"推荐"、"建议"、"最适合"、"最佳选择"、"成功率 60%"、"现在申请"、"抓紧时机"等表达。（允许的表达形态有待进一步讨论后写入本条。在讨论完成前，以 §6.4 与本条禁用清单为底线。）

3.7.3 "**我们提供信息，不提供决定**" 必须在产品首屏、付费页、路径详情页各一处显式可见。不得埋进服务条款。

3.7.4 产品不对用户的移民结果负责。此立场不是免责声明话术，是产品形态承诺——因此永远不做"按我说的做就能拿到 X"类保障/代办/对赌业务。

3.8 **国家入选的一手源充分性要求**。一个国家能否进入目标国集合，除满足 §3.5 的用户画像适配 + 合法专业人才路径可达之外，还必须满足：该国政府公开披露了产品字段清单所需的一手数据——至少包括签证 / 永居 / 入籍规则、按职业或学历或经验分层的收入披露、个人所得税税表、通胀与基础消费者价格指数。任一项长期缺失即暂不纳入。此条同时排除一切"买身份"为主的小国（披露薄，亦不属于专业中产路径）。

## § 4 · 技术架构

4.1 **目录即边界**。数据脚本、产品应用、数据库 schema、管理后台各自独立，互不 import。数据抓取代码**不得**与 Web 应用放在同一目录树里。

4.2 **任何脚本不得直接写产品数据库的产品表或 `public/data/*` 类静态数据文件**。必须经由 `data/build.mjs`。CI 检查这一点（grep 检测脚本中的直接写入路径）。

4.3 **数据库 schema 先于 UI**。先把一条端到端的真实数据（含四元组）录完，再反推 schema，再反推 UI。禁止先画原型再回头塞数据。

4.4 **禁止"通用引擎"的诱惑**。凡是听起来可以"覆盖所有国家 × 所有职业 × 所有情形"的通用计算层，都必须被拆解为少量、精确、手写的规则。通用引擎的维护成本永远比写 N 条规则高，准确度永远更低。

4.5 **依赖最小化**。能不引入第三方库就不引入。任何 npm 包上线前要回答：它是否会在 3 年内仍被维护？是否替代了 50 行以内能自写的逻辑？

4.6 **测试先于炫技**。数据仲裁层、签证规则匹配、税/成本计算这三处必须有单元测试覆盖；CI 不通过不得合并。UI 动效、首页排版可以无测试。

4.7 **禁止引入任何"AI 填补缺失数据"的代码路径**。即使包名叫 `ai-enrich`、`smart-fill`、`auto-complete`，只要输入包含用户字段、输出写入产品数据表，默认拒绝合并。

4.8 **禁止在生产代码中硬编码任何数值**作为 fallback。数值只能从数据层来，数据层给 null 就展示"暂无"。

## § 5 · 政策规则维护（护城河）

5.1 **每条政策规则必须记录**：authority（机构全名）、sourceURL、lastFetchedAt、lastReviewedAt、rawQuote、snapshotRef（见 §2.7）、reviewerHandle。

5.2 **6 个月强制复核**。`lastReviewedAt` 距今 > 180 天的规则在 UI 上降级为"待复核"，不参与任何推荐排序。**复核周期不得延长**，不得以"规则没变"为由跳过——复核这个动作本身就是护城河。

每次复核的操作定义是 §2.7 的一次拉取 + 存档：按 §2.7.1 在新日期目录下新增一份快照，计算新 sha256 与上一份快照对比。sha256 相同视为"规则未变"（仍需 commit 新快照以推进 `lastReviewedAt`）；sha256 不同视为"官方有修订"，按 §5.4 追加 changelog 并触发 §7.3 的用户通知链路。**不得以"目测内容一致"跳过拉取与存档**——复核这个动作不是判断，是留痕。

5.3 **规则录入是产品工作，不是运营工作**。录入人需要同时懂目标用户心智、懂规则语义、懂中文翻译，不接受"外包给翻译公司"、"让律师助理填"这类稀释。

5.4 **规则变更必须留 diff**。每次修改追加一条 changelog：日期 / 源链接 / 原条款 / 新条款 / 对已有用户保存路径的影响。

5.5 **任何"概率 / 成功率带 / 预计时间线"必须可追溯**。不允许出现"大约 60%"这种无法点进去看到计算依据的数字。不能追溯就写 `null` 或用非数字化的定性标签（例如"历史批准率未公开"），不得造数字。

## § 6 · 设计与文案

6.1 **视觉调性**：编辑风、文本密集、克制留白；Serif 标题 / Sans 正文；桌面优先。参考对象是严肃长读物与研究报告。禁止"炫酷 dashboard"、渐变大卡片、Dynamic Island 风、情感驱动首页 copy。

6.2 **结论先行，渐进披露**。每页首屏一句人话结论，滚动展开证据，点击深入细节。3 秒内用户必须能回答"我在看什么"。

6.3 **每个数字后面跟一句 insight 翻译**。裸数字不上 UI。数字必须伴随对用户切身处境的解读。

6.4 **文案禁用**：
- 装饰性 emoji（产品内文案全局禁用；UGC 内用户自己写不限）
- "宜居之选"、"顶级"、"全球排行榜"、"一站式"等形容词堆砌
- "AI 驱动"、"AI 赋能"、"智能推荐"、"大模型支持"等营销话术
- 任何制造"快点决定"紧迫感的话术

6.5 **i18n 字符串只能从 i18n 源文件取**，不得凭记忆写。改翻译时逐语言核对语义完整性，原文每个关键词都要被译到；引用子指标名称必须从 i18n 实际值读。

6.6 **每次 UI 修改必须逐属性 diff**（字号、字重、颜色、padding、透明度、继承关系），不得"看起来差不多就提交"。

6.7 **反向宣称"我们不做什么"的文案优先级高于"我们做什么"**。差异化来自拒绝，不来自叠加。

## § 7 · 营收与商业边界

7.1 **付费从第一天就是计划的一部分**，不是未来可选项。"免费 + 广告"的模型会把产品动机扭曲成"让用户停留更久"，永远禁止。

7.2 **广告位禁令**。产品内永不接入任何广告联盟、信息流广告、"赞助内容"。

7.3 **商业模型：信息全免费，年费买的是时间维度上的同步权**。

所有信息（签证规则、积分表、收入中位数、税表、时间线、一手源链接）对任何访客免费、无登录墙、无付费墙。付费不解锁信息；付费前后看到的字段和数值完全一致。

年费（后期量级待定）买的是两件东西：
- **"我的路径"保存**：用户打过的分、选过的国、关注的规则，形成可持久化的决策快照
- **规则变动通知**：用户关注的规则在 §5.2 复核中发生实际变更时，收到一封写清"原条款 / 新条款 / 对你的影响"的邮件

禁止稀释方案：低价试用、微付费、分享解锁、邀请返现、积分解锁。禁止按付费状态划分信息可见度（禁止任何形式的付费墙）。

**两重对齐的意义**：信息全开放 = 对应 §3.7 "不替用户判断"（永不以信息隐藏制造决策压力）；年费 = 对应 §5.2 复核机制的对外输出（复核这个动作本身成为对付费用户的周期价值交付）。

7.4 **未来的任何转介 / 佣金合作**（如与律所、税务师、教育机构）必须满足：

7.4 **未来的任何转介 / 佣金合作**（如与律所、税务师、教育机构）必须满足：
- 仅对付费用户开放
- 佣金不参与路径排序
- UI 全程显式标注合作关系
- 合作协议书面化，不得口头约定

7.5 **B2B 数据授权在产品成熟度达到"对 C 端已可长期稳定服务"前不启动**，不被任何"大客户找上门"的邀约提前启动。规则库不成熟就卖出去，会放大错误。

## § 8 · 品牌与对外表达

8.1 **`relocate` 是仓库的内部工作代号，不是产品名**。在产品被真实验证可用之前一律以此作为代号，不预先注册域名 / 商标 / 社交账号锁定一个未经验证的名字。Phase 0 阶段不进行任何形式的品牌动作（名称、视觉、对外叙事）。若未来真以 `relocate` 为名发布，是顺其自然的结果，不是本条的预设。

8.2 **仓库目录名 / 代码里的常量名不等于品牌**。外部资产（域名、Logo、公众号、推文）与仓库内部命名必须解耦，任何"反正仓库都叫这个就用这个"的惰性拒绝。

8.3 **公开复盘是品牌资产**，但每一次公开复盘必须引用真实数字（见 §2.6），不得为叙事戏剧性添油加醋。戏剧性低一点的真相 > 戏剧性高一点的注水。

8.4 **不跟任何"润学"灰色语境绑定**。风控意识是产品能活下去的前提。

## § 9 · 路线图反模式

9.1 **Phase 0 的唯一目标**：在 §3.6 锁定的国籍前置下，打通 **日本 × 高度人材（HSP 积分制）** 这一条端到端流程。验收标准为**作者本人以产品为真实个人决策依据完整走完一次**（dogfood 验收），包括：输入条件 → 看到可行性/可负担性/回报周期三个输出 → 逐字段点开官方源核对 → 据此形成自用的行动清单。

此阶段**不做**：首页、i18n、AI 接入、空数据行、多路径扩展、任何形式的外部用户访谈/内测/招募/营销。

**此阶段产品不对外公开发布**（第三道防线）。验收通过前不绑定公开域名、不向任何非作者开放访问。系统性错误依靠三道防线共同防范：预防（§2/§5） + 免责（§3.7.4） + 延迟公开（本条）。万一在 dogfood 阶段仍出现系统性错误，不得静默修复；须在项目内变更日志（将来的 CHANGELOG 或 retrospectives）中记录错误性质、发生日、修复方式。

9.2 **第 3 个月还在抓数据 / 扩城市 / 打磨首页 = 走偏**。优先级永远是 `路径 > 覆盖面 > UI 精美`，不可倒置。v1–v3 的目标**不是国家数量**，而是"**稳定、可单人长期维护的国家数量**"。新增一国的前置条件是现有所有国家的 180 天复核无逾期、字段完整性无劣化。README 与本文件中提及的 ~15 国是**战略天花板**，不是短期覆盖目标。

9.3 **KPI 不得是"功能数"或"页面数"**。Phase 0 的北极星是 "作者本人基于产品输出形成自用行动清单" 这一动作是否真实发生；Phase 0 之后的北极星是付费用户完成一次完整决策流的次数。两个阶段都不采用 DAU / 停留时长这类注意力指标。

9.4 **每新增一条政策规则，录入人须逐字段对照官方一手源完成一次独立复核**（与首次录入由同一人在不同日期进行，或由第二人完成）。未经独立复核的规则不得参与产品展示。

9.5 **如果 3 个月后产品还没有任何付费用户，问题出在定位，不在功能**。回到 §3 重读，不要再加功能。

9.6 **阶段结束失败自审清单**。每个 Phase 结束当天，负责人必须对以下清单逐条书面自审（写入项目内 retrospectives 记录），不得以"感觉还行"跳过。

**硬失败**（触发任一 = 需要重新立项，不是打补丁）：
- F1. 本阶段的 dogfood 完成后，自己仍然回去小红书 / AI 助手 / 中介复核关键答案 —— 产品只是加了一层，没替代任何东西
- F2. dogfood 完成后，说不出"产品帮我做了某个原本做不到的动作"，只能说"信息全、诚实" —— 产品只是信息库，没到决策引擎（最隐蔽的失败）
- F5. 首个 180 天强制复核到期时无动力完成，或草草过 —— §5.2 的护城河已经空转
- F8. 付费上线后 3 个月，没有一位非熟人、非主动推介的用户付费 —— T1 的定位或对手判断错了

**软失败**（单个 Phase 内触发 2 项 = 视同硬失败）：
- F3. 用户决策的某个关键字段为 null 且无对应一手源可补 —— 回到 §2.1.4 审视该字段是否本就不该存在
- F4. 新增第二条路径时发现 Phase 0 的 schema / 流程 / UI 需推倒重写 —— 方法论未沉淀
- F6. 为"界面好看"动念用估算值补 null —— §2.2.2 在现实压力下失效
- F7. 每新增一国的边际人力 / 时间成本未下降 —— 在做重复劳动，无护城河积累

## § 10 · 违反处置

10.1 **违反 §2 任一条 = 立刻回滚 + `data/retirements.log` 留痕 + 必要时 UI 下线相关字段**。不讨论、不降级处理。

10.2 **违反 §3.3（永久不做清单）= 立刻关闭分支**。需要重新立项并修改本 RULES.md 才能重启。

10.3 **违反 §6 = 该页不上线**。有疑问的改动默认不合并。

10.4 **连续违反同一条规则 3 次以上 = 负责人必须写一份公开复盘**，作为信任代价公开发布。
