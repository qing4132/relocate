§ 1把 Phase 0 的硬约束摆在桌上

在画任何页之前，先把 RULES 里所有跟"网站长什么样"直接相关的硬约束抓出来——它们决定我们能做什么页、不能做什么页。

从 README 和 RULES 中过出来的、与信息架构直接绑死的约束有七条：

1.1Phase 0 不做首页

RULES § 9.1 列举本阶段不做：首页、i18n、AI 接入、空数据行、多路径扩展……。这是我们最容易忽略的一条——"不做首页"意味着我们没有营销首屏、没有"产品介绍 → CTA"的常规漏斗，用户进站就直接落到一条具体路径。Phase 0 站内合法的页只有"路径本身"和"路径相关的页"。

1.2验收动作就是"四步走"

RULES § 9.1 把 Phase 0 验收定义为：输入条件 → 看到三输出 → 逐字段点开官方源核对 → 形成自用行动清单。这四步本身就是页面骨架——网站至少要支撑这四个动作的连贯发生。

1.3三件输出不变

README 锁定了输出三件：可行性 / 可负担性 / 回报周期。任何页都不得引入第四个维度（"教育"、"医疗"、"幸福指数"等都属于 § 3.1 拒绝的"再加一个比较维度"）。

1.4付费前后字段一致

RULES § 7.3 的"信息全免费 + 年费买'我的路径'保存 + 规则变动通知"决定：付费状态只影响"是否能保存 / 是否能收通知"，不影响任何字段是否可见。换言之，**网站没有"游客版"和"会员版"两套页面**，只有"已登录看到了一个'保存'按钮"和"未登录没看到"的差别。

1.5"我们提供信息，不提供决定"必须三处可见

RULES § 3.7.3 要求这句话在首屏 / 付费页 / 路径详情页各一处显式可见。Phase 0 没有首页，于是这句话现阶段只能落在：路径详情页 + 测算入口处 + （未来的）付费页。

1.6每个数字必须可点开看源

RULES § 2.4.1 要求每个对外展示的数值，必须在至少一个可达交互里暴露 sourceURL + fetchedAt。这意味着站内必须存在一类"字段溯源"的可达终点——可能是单独的字段详情页，也可能是路径页内的 popover，但不能没有。

1.7结论先行，渐进披露

RULES § 6.2 要求每页首屏一句人话结论。这条决定我们的页内结构：所有页都是"结论 → 证据 → 详情"的纵向序列，不是 dashboard 的横向并列。

六个站内页 + 入口的逐页规格。每一页都标注了对应 Phase——Phase 0 必做的与可推迟的。

P0 · entry
/ · 入口（路径列表）Phase 0
目的
让用户在 3 秒内知道"这是什么、可走的路径有几条"。
受众
所有访客。
看到
项目代号一行；下方 1 张路径卡（jp-hsp-1-b-engineer）；再下方"暂未覆盖：14 个目标国"的体面表达（§ 3.4）；footer 的 about 入口。
不看到

§ 4关键回答 · 通用说明 vs 个性化结果

你最直接的那个问题：这页要不要是详尽通用说明？还是个性化定制？答：两者都要，但它们是同一篇文章的两个实例。

P1 · 路径介绍页

主语是"这条路径"。用 Alice 这个示例人物穿在叙述里，作为解释规则的工具。任何访客看到的文字相同，包括"以 Alice 为例她拿了 70 分"这种段落。

P4 · 测算结果页

主语是"你"。叙述结构、章节顺序、引用与脚注完全沿用 P1，但凡 Alice 出现的地方都被换成"你"，凡示例数字都被换成你的输入。是 P1 的实例化渲染，不是新页面。

这样做的三个好处：

第一，避免双重维护——只写一份文章模板，介绍页与结果页同源。第二，避免认知割裂——读完 P1 想算自己，跳到 P4 看到完全一样的章节结构，没有"重新学习一份新页面"的成本。第三，避免黑箱感——你看到自己的得分时，每一节都对照着规则解释为什么是这一档，跟 P1 上的写法是同一套语言。这正是你那条"我们没有黑箱、我们透明"的具体落地。

§ 5入口长什么样

RULES 不让我们做首页，但用户进站总要先看到点什么。我们用最朴素的形态承担这个职责——一份"路径目录"。

入口（P0）的最小内容只有四块：

① 顶部一行：项目代号 + 当前 RULES 版本日期。② 一句话说明："中文母语技术移民申请人的合法路径检索。"——这是替代"hero copy"的一句人话，不带任何形容词、不带 CTA。③ 路径目录：当前唯一一张卡 jp-hsp-1-b-engineer，副标题"日本 · 高度専門職 1 号ロ · 技術系"，再附一行"最近一次复核 2026-04-10"。④ 暂未覆盖区：列出我们承认能力天花板内的另外 14 个目标国名（不是无限多个、不是按"全球 200 国"假装）；每个都标"暂未覆盖"，不允许 hover 出"敬请期待"。

没有搜索框（一条路径不需要搜索），没有筛选器（同样原因），没有"个性化推荐"（§ 3.7 拒绝）。Phase 1 增加路径数量后，这一页演化为可筛选目录——但筛选维度永远不超过"目标国"和"用户画像类别"两个，绝不引入"宜居度"、"幸福指数"、"AI 推荐"。

§ 6需要你拍板的开放问题

下面 5 个问题是这一版方案不能单独由我决定的——任何一个的答案都会改动 § 2 站点地图。

  1. P3 测算页是必须的吗？另一种方案是干脆把"测算"做进 P1——用一个 sticky 侧栏让你边读边填，所有计算实时反映在文中。这样就没有 P3 → P4 的跳转，整个网站收敛为 4 个页（P0/P1/P2/P5/P6 减一个）。代价是 P1 变重，移动端不友好。
  2. P4 是独立 URL，还是 P1 的 query string 实例？如果是后者（如 /paths/.../?as=you&age=34），则不需要 P4 这个独立路由，"通用 vs 个性化"是同一 URL 上的两个状态。这种做法更彻底，但分享链接时要标得很清楚。
  3. P5 changelog 是路径维度还是字段维度？路径维度是"jp-hsp-1-b-engineer 的所有变更按时间排"；字段维度是"年薪 × 年龄表的所有变更按时间排"。两者都对，但 IA 不一样。
  4. "暂未覆盖"的 14 国列表，从哪儿来？RULES § 9.2 提到 ~15 国是"战略天花板，不是短期覆盖目标"。但具体哪 15 国，至今没有写进 README/RULES。是否需要我先列一份候选清单与你过一遍，落进 SOURCES.md？
  5. P6 是 1 页长文，还是分成"章程 / 数据原则 / 我们不做什么"三页？三页更便于检索，1 页更便于一次读完。
