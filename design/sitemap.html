§ 1用户心智的原话

承袭 v0.2 §1。从用户脑中的话出发，不是从产品功能出发。

我想离开中国，可是去哪呢？哪里好？哪里适合我？哪里我能去呢？

（或者是另一个我：我就是要去美国。）

怎么去？需要什么？要做哪些准备？

我现在能去吗？我三年后能去吗？如果我现在不能去而三年后可能能去，我在这三年里要付出什么努力？

有没有什么硬约束？我最好走哪条路径？别的国家我也想了解一下。

  1. "我想走，可是去哪？" M01 · 探索：没目标，要全局看一眼。
  2. "我就是要去美国 / 德国 / 日本。" M02 · 锁定国家。
  3. "怎么去？要做哪些准备？" M03 · 沉到具体路径看规则。
  4. "现在能去吗？三年后呢？要付出什么？" M04 · 现状 vs 阈值，差距与改造路径。
  5. "有没有硬约束？" M05 · 出境侧 / 入境侧硬前置。
  6. "别的国家我也想看，自己比较一下。" M06 · 横向对照。
  7. "我心里有数。" M07 · 数据从哪来 / 没数据的诚实承认。

§ 2核心抽象 · PathCard

把"路径"提升为整个站点唯一的一等公民。每个 PathCard 是一个具有稳定字段的对象；网站的几乎所有列表与详情都是它的不同视图。

这层抽象同时解决三件事：

第一，降低用户心智负担——无论用户从"探索""国家页""目录""比较"哪条路进来，看到的都是同样形状的卡片，只是上下文与筛选不同。视觉一致 = 学习成本一次性。

第二，压缩 IA——v0.2 里 P_explore / P_index / P_country 看似是三个不同页，本质上都是 PathCardListView + 不同 filter 预设。我们只设计一个组件，三处复用。

第三，对齐 RULES § 3.7——卡片只是摆事实（这条路径有什么属性、当前用户与门槛的相对位置），不打分、不排序、不推荐。filter 是机械的，"对你够格 / 差几步 / 远不够"是布尔判定，不是产品意见。

2.1PathCard 的字段

每个 PathCard 由两类字段构成：展示字段（用户能看到、出现在卡片正面）和索引字段（不一定显示，但可被 filter 引用）。所有字段都受 RULES § 2.1 约束——必须有一手源，禁止估算。

字段类型用途
idstring稳定标识符，如 jp-hsp-1-b-engineer永不变。URL、引用、changelog 都靠它。
countryISO-3166-1 alpha-2所属国家，如 jp。filter 维度 · 国家。
namei18n string路径中文名 + 官方原文名。"日本 · 高度専門職 1 号ロ · 技術系"。
categoryenum路径类别：skilled / investor / family / national 等。v0 只覆盖 skilled，其余卡片可存在但标"暂未覆盖"。
summarytext · 一句话卡片正面那一行说明。不是营销话术，是规则的中性概括。
fitProfileenum[]路径适配的画像标签：tech / research / finance / medical 等。filter 维度 · 适配人群。
thresholds结构化路径硬阈值，如 HSP 的 {score: 70} 与 {score: 80} 两档。驱动"够格 / 差几步 / 远不够"的布尔判定。
permanentResidencyMonthsint / range从获签到可申请永居的最短月数。filter 维度 · 时限（如"5 年内可永居"）。
citizenshipMonthsint / range到可申请入籍的最短月数。同上。
coverageStatusenumcovered / uncovered / retiredv0 唯一一张是 covered，其余明列 uncovered。filter 默认包含 uncovered，体现 § 3.4 的诚实。
lastReviewedAtISO-8601 dateRULES § 5.1 必备。超 180 天降级"待复核"。filter 维度 · 数据新鲜度。
sourcesRefstring[]该路径所引用的所有一手源 docId 列表。是 P_source 与 changelog 的反向索引。

2.3filter 是产品的中枢操作

一旦 PathCard 字段完整，所有"列表型"页面都退化成同一个组件 PathCardListView(filter, sort, context) 的不同实例。三个参数：

filter　是字段上的布尔/范围谓词组合。例：country = 'us'、category = 'skilled'、permanentResidencyMonths ≤ 60、fitProfile ∈ {tech}、coverageStatus = 'covered'。多个 filter 之间默认 AND，同字段多值 OR。

sort　Phase 0 不引入。RULES § 3.7 不替用户排序——任何"按推荐度排"都是产品意见。退而求其次允许的中性排序：按国名字典序、按复核日期倒序。除此以外不开放。

context　是当前用户的属性快照（来自 P_explore 的输入或 URL query string）。卡片用它来计算"对你的判定"那一行（够格 / 差几步 / 远不够）。无 context 时不显示这一行。

§ 4视图与页面规格

区分两个层级：组件（PathCardListView / PathDetailView）和页面（具体 URL）。组件可被多页复用，页面是 URL × filter 预设的命名实例。

4.1组件层

CMP-1
PathCard
责任
呈现一个路径的稳定卡片视图。Properties: path (PathCard object) 与可选的 context (user attribute snapshot)。
状态
四态：covered+context、covered+无 context、uncovered、retired。视觉差异由 CSS 修饰类控制。
不做
不打分、不排序、不带"推荐"标识。卡片是事实容器，不是产品意见容器。
CMP-2
PathCardListView
责任
展示一组 PathCard。Properties: filter、sort、context。
头部
当前 filter 摘要（"国家 · 日本 / 类别 · 技术人才 / 你目前可走的"）；filter 编辑器（增删条件、不带"重置为推荐"按钮）。
列表
纵向排列的 PathCard。已覆盖卡在前，暂未覆盖在后；同组内按字典序。
空态
filter 命中 0 张时显示"当前条件下，本产品覆盖范围内无路径符合"——不显示"试试放宽筛选"，因为那是产品在替用户判断"应该放宽"。
CMP-3
PathDetailView
责任
呈现一条路径的全部叙述与规则。Properties: path、可选 context。
章节
§1 这是什么 / §2 打分表逐项（带 [n] 角标）/ §3 阈值含义 / §4 永居入籍时间表 / §5 一句"我们提供信息不提供决定"/ §6 "看怎么补 / 看硬约束" / §7 引用与快照。
个性化
有 context 时，叙述里的"以 Alice 为例"段落被替换为"以你为例"，但章节结构与文字不变。同一篇文章的两个实例。

§ 5抽象的代价 · 必须自陈

每一层抽象都有代价。这里把 PathCard 抽象不能解决的问题写下，避免下一轮再被问出。

① 路径的"故事感"会被卡片磨平吗？

会，如果只看 /list。但用户从卡片点进去后到达的 PathDetailView 仍是衬线长读物——故事感留在详情层，列表层负责扫视。这是"概览统一 / 详情个性"的常见取舍。

② 不同路径字段不一定同构（如积分制 vs 抽签制）

美国 H-1B / EB 这类没有"积分阈值"的路径，thresholds 字段会是 null 或换成"抽签率 / 排期年数"。PathCard 的字段集需要做差异分支而非"假装统一"。这要求我们在 schema 上引入type 字段（scoring / lottery / queued / investor / …），不同 type 有不同的子字段集。Phase 0 只覆盖 scoring，但 schema 必须为其他 type 留口子，否则 Phase 1 加美国时整个数据层重写。

③ "适合谁"的 fitProfile 标签由谁定义？

这是隐藏的产品意见。tech / research / finance / medical 这种粗分类还行，但一旦细分到"前端工程师"或"半导体工艺工程师"，就会变成产品在替日本入管局的职业列表做翻译——RULES § 2.1.4 警告过这种行为。Phase 0 标签必须保持极粗，宁可让用户感到"分得不细"，也不要让用户感到"我们替他做了职业判断"。

④ filter 的字段集 = schema 的子集

这点反过来反推 schema 设计：任何"用户可能想筛"的维度，背后都必须有对应字段，且字段必须满足 RULES § 2.1（一手源、五元组、有快照）。"是否对中国大陆护照宽松"这种维度，没有现成一手源量化，就不能成为 filter——这是抽象的硬边界，不是 UI 的偷懒。

§ 6需要你拍板的开放问题

  1. PathCard 的字段集 v0 锁哪几个？我列了 11 个。Phase 0 只覆盖 scoring type 的日本 HSP 一条；其余字段必须可空、可缺。最危险的是 fitProfile——它的标签集一旦写进 i18n 文件就难改。建议 v0 只引入 tech、research、medical、finance、arts、other 六个，宁少勿多。
  2. filter 维度对外开放哪几个？建议 v0 开放：country、category、coverageStatus、permanentResidencyMonths（"几年内可永居"）四个。fitProfile 不在 v0 开放（避免上一条的风险）。
  3. "对你的判定"是否进 v0？它要求实现 thresholds 比对引擎。Phase 0 只有一条路径，引擎会非常薄；但一旦引入，未来添加 lottery type 的路径时它要泛化到"抽签率与你画像的关系"。是否把这件事放到 Phase 1？放的话 v0 的 PathCard 在无 context 下也能用，缺一个用户最想看的卖点。
  4. P_plan 的"改造路径"是否复用 PathCard 视觉？建议复用骨架不复用组件——视觉一致让用户更容易理解，但底层数据模型是 Patch object（"考 N1 +5 分 · 6-12 月 · ¥10-30k"）而非 Path object，不应混到同一个类里。
  5. ~15 国战略天花板的具体清单承袭 v0.2 §7 Q4。PathCard 抽象使这件事更迫切——/list 一上线，所有 uncovered 卡片都要有数据（哪怕只有 country / name / coverageStatus）。
