§ 1用户心智的原话

先把用户进站时脑子里的话原样写下。这是 IA 的唯一锚点。

我想离开中国，可是去哪呢？哪里好？哪里适合我？哪里我能去呢？

（或者是另一个我：我就是要去美国。）

怎么去？需要什么？要做哪些准备？

我现在能去吗？我三年后能去吗？如果我现在不能去而三年后可能能去，我在这三年里要付出什么努力？找个跨国企业？读研？考雅思？学日语？还是什么？

有没有什么硬约束（比如某些公务员岗位会收护照之类的）？

我最好走哪条路径？别的国家我也想了解一下，我也想自己判断一下，看看自己适合哪里，万一我觉得自己更适合德国呢？

这段话里至少有 七个不同的心智节点，它们出现的顺序就是用户在站内移动的顺序。我们的 IA 不是去"教育"用户先做哪一步，而是让用户在每一个节点上都能找到下一个自然落点。

§ 2把心智拆成节点

每一节点对应"用户在那一刻想干的一件事"。后面的页面规格直接服务这些节点，而不是服务"产品逻辑"。

  1. "我想走，可是去哪？" 探索阶段。用户没有目标国，希望被引导：从他自己的几个属性出发，看哪些国家"够格 / 差一点 / 远不够"。 → P_explore
  2. "我就是要去美国 / 德国 / 日本。" 目标已定，要直接看那一国的整体图景：有几条专业人才路径、各自规则有多苛、自己大致落在哪。 → P_country
  3. "怎么去？需要什么？要做哪些准备？" 沉到一条具体路径里看规则全文：分数怎么打、阈值是什么、永居和入籍如何衔接。 → P_path
  4. "我现在能去吗？三年后能去吗？这三年我要付出什么？" 当前现状 vs 阈值，差距是什么；填补每一条差距各自需要的钱、时间、动作（学语言、考资格、跨年薪档、读研）。 → P_plan（新增 · 这是过去缺失的核心）
  5. "有没有硬约束？" 不是签证规则，而是"想申请/不能申请"的前置：护照管控（现役军公教 / 涉密岗位）、特定行业、特定经历。这一类我国一手源不一定好找，要诚实标"暂未覆盖"或"建议去某某查"。 → P_constraint（新增）
  6. "别的国家我也想看一下，我想自己比较。" 从一国跳隔壁国，或者把两条不同国的路径对照着读。注意：不是"打分排序推荐"，是"把同一套字段的两份原文摆在一起，让用户自己看"。 → P_country / P_path 的横向链接 + 比较视图（Phase 1+）
  7. "我心里有数了。" 哪怕我们答不全（很多问题超出 v0 能力），也要让用户知道——"这件事我们没数据，建议去 X 查"。RULES § 3.4 已经保护了这一点。 → 渗透在所有页的"暂未覆盖"显式表达

三个观察：

第一，没有"主页"，但有"入口"——入口承担三件事：交代我们做什么、给三种入路、收尾。RULES § 9.1 不让做"营销首页"，本入口只做导航。

第二，P_country 是真正的"枢纽"——所有路径都挂在它下面，一切横跳也以它为中转。Phase 0 只点亮一国，但骨架按"将来 N 国"设计。

第三，P_plan 是过去几版方案没有的——它直接对应你那条 "三年里要付出什么"，是网站从"匹配工具"走向"决策引擎"的关键页。

§ 4逐页规格

八个站内页 + 一个 Phase 1 占位。每一页都直接服务 § 2 中的某个心智节点，不绕。

P_plan
/paths/<id>/plan · 准备计划Phase 0 · 关键新增
服务心智
M04 — 我现在能去吗？三年后能去吗？要付出什么？
看到
用户输入自己的当前条件（与 P_explore 共享一份 schema）；按规则逐条比对，给出"你当前在哪一档、距下一档差多少、可以通过哪些动作填补"——每条改造路径独立列，标注预计成本（钱）、预计时长（月）、外溢副作用（如读研换签证类型）。
关键
这是把"决策引擎"从口号落到产品形态的页。RULES § 3.1 说"输出可执行的路径 + 概率 + 时间线 + 真实成本"——P_plan 是它的物化。同样，绝不替用户挑哪一条改造路，只把所有可改造路摊开。
风险
"填补成本"本身可能涉及第二层数据（语言学校学费 / JLPT 考试时长 / 国内研究生学费）——这些数据是否能找到一手源、是否进 schema，要按 RULES § 2.1.4 严判。Phase 0 可以只点亮"语言考试"和"年薪跨档"两条最容易找源的，其余明示"暂未覆盖"。
出口
每条改造路径 → 一个详情子页（Phase 1+，先用占位）；返回 P_path。
P_constraint
/paths/<id>/constraints · 准入硬约束Phase 0 · 关键新增
服务心智
M05 — 公务员收护照之类的硬前置。
看到
分两类：出境侧（中国大陆护照管控、敏感行业、政审、户籍等）与入境侧（目标国对申请人的政治、犯罪、健康审查）。每一条都标"我国一手源 / 目标国一手源 / 暂未覆盖"。
关键
这是最容易"暂未覆盖"的一页——出境侧的相关规定散落在多个部门规章、地方性文件、内部规定里，未必有公开稳定一手源。诚实地说"我们查不到 / 这块你需要去 X 部门官网或问律师"，比塞东西强。

RULES § 8.4 不跟"润学"灰色语境绑定——这一页措辞要尤其克制，只摆事实，不指导规避。

§ 5张力：用户想比较，Phase 0 只有一国

这是新方案最大的诚实点：用户心智天然要比较，但我们 Phase 0 只能点亮日本一条路径。这两件事如何共存？

取舍

IA 骨架按"未来覆盖 N 国"设计——P_explore 的表、P_index 的目录、P_country 的枢纽位置都是为多国预留的。Phase 0 不假装多国，而是在每一处显式列出"暂未覆盖"的国家清单（§ 9.2 提到的战略天花板的 ~15 国）。

这样做的代价是 Phase 0 上线时网站看上去 "很空"——大半个 P_explore 表、大半个 P_index 目录都是"暂未覆盖"灰行。但这就是事实。RULES § 3.4 已经决定了"明确说我们还没做比假装覆盖更体面"。

这样做的好处是 IA 不会在 Phase 1 加第二国时被迫推倒重写——这正是 § 9.6 失败清单 F4 想避免的事。

§ 6这版与 v0.1 的关键变化

1. 入口从"路径列表"改为"三种入路"——v0.1 假设用户已经选定要看路径，v0.2 承认"还没想好"是合法的初始状态。

2. 增加 P_explore——直接服务"我想走，去哪"那一句心智。这是 v0.1 缺失的最大节点。

3. 引入 P_country 作为枢纽——v0.1 把"路径"作为唯一一级，但用户脑中"国家"先于"路径"存在。

4. 增加 P_plan——直接服务"三年后能去吗、要付出什么"。这是网站从"信息库"走向"决策引擎"的关键，也是过去几版被你诟病"沦为查询工具"的解法。

5. 增加 P_constraint——直接服务"公务员收护照"那条。诚实地把出境侧硬约束摆出来，哪怕大量"暂未覆盖"。

6. 横向跳转作为一等公民——P_country / P_path 底栏都能跳隔壁国家，对应"万一我更适合德国"那条心智。

7. P_path 不再是"事实首页"——它退到第三层，是用户沉到具体路径里看规则全文的页，不再承担"承接所有访客"的职责。

§ 7需要你拍板的开放问题

  1. Phase 0 是否实现 P_explore 与 P_index？这两个页骨架完整需要假设"未来 ~15 国清单"已确定。如果 Phase 0 只想以 P_country/jp + P_path/...（+ 必要末梢）上线，可以推迟到 Phase 1。但代价是 Phase 0 dogfood 期间没有"探索入路"。
  2. P_plan 的"改造路径"在 Phase 0 里点亮几条？我建议只点亮两条最有一手源支撑的——"考 N1 日语 +5 分"与"跨年薪档 +X 分"。其余（读研、考资格）显式"暂未覆盖"。这样不破 RULES § 2.1.4。
  3. P_constraint 的出境侧要不要做？如果做，要直面"我国一手源稀缺、可能大段空白、措辞要克制"的挑战。如果不做，至少要在 P_path 里加一行"出境侧准入约束不在本产品覆盖范围"——把缺席本身明示。
  4. ~15 国战略天花板的具体清单，是否要先开一轮独立讨论敲定？这清单一旦确定，会出现在 P_explore 和 P_index 的"暂未覆盖"列表里，意味着我们对外承诺"这些国未来会做"。
  5. "暂未覆盖"是否进入产品 schema？也就是说，是否在数据层为每一条未覆盖路径预留一条记录（含"未覆盖原因"字段）。如果是，schema 就需要从 Phase 0 起就支持"路径状态"枚举。如果不是，UI 上的灰行只是 hardcoded 的列表，schema 不动。
